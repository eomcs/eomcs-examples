# app50. 리액트 성능 최적화 (심화)

## 개요

이번 장은 리액트의 성능 개선을 학습하기 위한 목적으로만 다룬다. 리액트는 기본적으로 빠르게 동작하므로, 대부분의 리액트 애플리케이션에서는 별도의 최적화가 필요하지 않다. 자바스크립트와 리액트를 위한 좀 더 정교한 성능 측정 도구들도 있지만, 여기서는 간단한 `console.log()`와 브라우저 개발자 도구의 로깅 결과만으로 살펴본다.

> Java에 비유하면, "대부분의 경우 최적화가 필요 없다"는 태도는 자바에서 JIT 컴파일러와 JVM이 기본적으로 상당히 잘 최적화해주기 때문에, 프로파일러로 실제 병목을 확인하기 전까지는 섣불리 마이크로 최적화를 하지 말라는 조언과 일맥상통한다. 리액트의 `StrictMode` 역시 자바의 `-Xlint`나 정적 분석 도구처럼, 실제 운영 환경에는 포함되지 않고 개발 단계에서만 잠재적인 버그(부작용의 오용 등)를 표면화시켜주는 도구다.

## Strict Mode

리액트의 성능을 배우기에 앞서, 먼저 `src/main.jsx` 파일에서 활성화되는 리액트의 **Strict Mode**를 잠깐 살펴본다.

```jsx
// src/main.jsx
createRoot(document.getElementById('root')).render(
  <StrictMode>
    <App />
  </StrictMode>
);
```

리액트의 Strict Mode는 구현에 문제가 있을 때 개발자에게 알려주는 헬퍼 컴포넌트다. 예를 들어 지원 중단(deprecated)된 리액트 API(레거시 훅 등)를 사용하면 브라우저 개발자 도구에 경고가 나타난다. 뿐만 아니라 개발자가 상태와 부작용(side-effect)을 제대로 구현했는지도 함께 검증해준다. 실제로 어떤 의미인지 코드로 확인해보자.

## 실습

### 실습 1. `useEffect`가 몇 번 실행되는지 확인하기

`App` 컴포넌트는 처음에 원격 API에서 데이터를 가져와 목록으로 보여준다. 이 데이터 페칭을 시작하기 위해 리액트의 `useEffect` 훅을 사용하고 있다. 이 훅이 실행될 때마다 로그를 남기는 `console.log()`를 추가해보자.

```jsx
// src/App.jsx
const App = () => {
  // ...

  React.useEffect(() => {
    console.log('How many times do I log?');
    handleFetchStories();
  }, [handleFetchStories]);

  // ...
};
```

대부분은 이 부작용이 한 번만(또는 `handleFetchStories` 함수가 바뀔 때만) 실행돼야 하므로 브라우저 개발자 도구에 로그가 한 번만 찍힐 것으로 기대한다. 하지만 실제로는 `App` 컴포넌트의 초기 렌더링에서 로그가 두 번 찍히는 것을 볼 수 있다. 솔직히 말해 이는(경험 많은 리액트 개발자에게도) 상당히 예상 밖의 동작이라서, 리액트를 처음 배우는 사람에게는 이해하기 어려울 수 있다. 그럼에도 리액트 코어 팀은 부작용이 잘못 사용된 애플리케이션의 버그를 표면화하기 위해 이 동작이 필요하다고 판단했다.

즉, 리액트의 **Strict Mode**는 **초기 렌더링에서 `useEffect` 훅을 두 번 실행한다.** 이로 인해 같은 데이터를 두 번 가져오게 되지만, 우리 애플리케이션 입장에서는 문제가 되지 않는다. 이런 동작을 **멱등(idempotent)** 하다고 부르는데, **요청이 성공적으로 수행된 결과가 그 요청이 실행된 횟수와 무관하다는 뜻이다.** 결국 네트워크 요청이 두 번 발생한다는 점에서 성능상의 문제일 뿐, 애플리케이션이 버그가 있는 동작을 하는 것은 아니다. 게다가 **Strict Mode는 개발 환경에서만 적용** 되므로, 애플리케이션을 프로덕션용으로 빌드하면 Strict Mode는 자동으로 제거된다.

이처럼 초기 렌더링에서 `useEffect` 훅이 두 번 실행되는 점과, 개발 환경과 프로덕션 환경의 동작 결과가 다르다는 점은 리액트의 Strict Mode를 둘러싼 여러 논쟁을 불러일으킨다.

### 실습 2. 이어지는 성능 실습을 위해 Strict Mode 잠시 제거하기

앞으로 이어질 성능 관련 실습들에서는, Strict Mode를 잠시 제거해서 애플리케이션이 프로덕션용으로 빌드됐을 때와 동일한 로깅 결과를 따라가 보기를 권한다.

```jsx
// src/main.jsx
createRoot(document.getElementById('root')).render(<App />);
```

다만 성능 관련 실습들을 마친 뒤에는, Strict Mode가 결국 개발자를 돕기 위해 존재하는 도구이므로 다시 추가해두는 것이 좋다.

## 첫 렌더링에서는 실행하지 않기

이전에 부작용(side-effect)을 위해 사용하는 리액트의 `useEffect` 훅을 다뤘다. 이 훅은 컴포넌트가 처음 렌더링(마운트)될 때 한 번 실행되고, 이후 리렌더링(업데이트)될 때마다 다시 실행된다. 두 번째 인자로 빈 의존성 배열을 전달하면 훅이 첫 렌더링에서만 실행되도록 만들 수 있다. 하지만 기본적으로는 첫 렌더링(마운트)에서는 실행하지 않고 리렌더링(업데이트)에서만 실행하도록 지시하는 방법은 따로 제공되지 않는다.

예를 들어, 로컬 스토리지를 이용한 준영속 상태 관리를 위해 `useState`와 `useEffect`를 함께 사용하는 커스텀 훅 `useStorageState`를 살펴보자.

```jsx
// src/App.jsx
const useStorageState = (key, initialState) => {
  const [value, setValue] = React.useState(
    localStorage.getItem(key) || initialState
  );

  React.useEffect(() => {
    console.log('A');
    localStorage.setItem(key, value);
  }, [value, key]);

  return [value, setValue];
};
```

개발자 도구를 자세히 보면, 이 커스텀 훅을 사용하는 컴포넌트가 렌더링될 때 처음부터 로그가 찍히는 것을 볼 수 있다. 그런데 컴포넌트가 처음 렌더링될 때는 초기값 외에 로컬 스토리지에 저장할 것이 없으므로, 이 시점에 부작용을 실행하는 것은 사실 의미가 없다. 이는 불필요한 함수 호출이며, 컴포넌트가 업데이트(리렌더링)될 때만 실행돼야 한다.

앞서 말했듯, 리액트에는 리렌더링마다 실행되는 훅이 따로 없고, `useEffect` 훅에 "리렌더링 할 때만 함수를 호출하라"고 지시할 방법도 없다. 하지만 리렌더링 사이에도 `ref.current` 프로퍼티를 그대로 유지하는 `useRef` 훅을 이용하면, 컴포넌트의 생명주기 동안 상태 갱신으로 인한 리렌더링을 유발하지 않는 인스턴스 변수를 만들 수 있다.

### 실습 1. `useRef`로 마운트 여부 추적하기

```jsx
// src/App.jsx
const useStorageState = (key, initialState) => {
  const isMounted = React.useRef(false);

  const [value, setValue] = React.useState(
    localStorage.getItem(key) || initialState
  );

  React.useEffect(() => {
    if (!isMounted.current) { 
      isMounted.current = true;
    } else {
      console.log('A');
      localStorage.setItem(key, value);
    }
  }, [value, key]);

  return [value, setValue];
};
```

- 컴포넌트가 처음 렌더링될 때는 useEffect 훅의 부작용 함수가 실행되지 않고,
- 리렌더링할 때부터 부작용 함수가 실행되도록 만들었다.

여기서는 리렌더링을 일으키지 않는 명령형(imperative) 상태 관리를 위해 `ref`와 그 가변(mutable) `current` 프로퍼티를 활용하고 있다. 컴포넌트에서 이 훅이 처음 호출될 때(컴포넌트 첫 렌더링), `ref`의 `current` 프로퍼티는 `isMounted`라는 이름의 `false` 불리언 값으로 초기화된다. 그 결과 `useEffect`의 부작용 함수는 실행되지 않고, `isMounted` 플래그만 부작용 안에서 `true`로 바뀐다. 이후 훅이 다시 실행될 때마다(컴포넌트 리렌더링), 부작용 안에서 이 불리언 플래그를 검사한다. 이제 값이 `true`이므로 부작용 함수가 실행된다. **컴포넌트의 생명주기 동안 `isMounted` 불리언 값은 계속 `true`로 유지되며, 이 값은 이 커스텀 훅을 사용하는 컴포넌트의 첫 렌더링에서 부작용 함수가 호출되는 것을 막기 위해 존재한다.**

지금까지는 컴포넌트가 처음 렌더링될 때 간단한 함수 하나의 호출을 막는 것에 불과했다. 하지만 부작용 안에 비용이 큰 연산이 있거나, 이 커스텀 훅이 애플리케이션에서 자주 사용된다고 상상해보자. 이런 경우에는 불필요한 함수 호출을 피하기 위해 이 기법을 적용하는 것이 훨씬 실용적이다.

## 필요하지 않다면 리렌더링하지 않기

앞서 리액트의 리렌더링 메커니즘을 살펴본 적이 있다. 이번에는 `App`과 `List` 컴포넌트를 대상으로 이 실험을 다시 해본다.

### 실습 2. `App`과 `List`에 로깅 문 추가하기

두 컴포넌트에 각각 로깅 문을 추가한다.

```jsx
// src/App.jsx
const App = () => {
  // ...

  console.log('B:App');

  return ( /* ... */ );
};

const List = ({ list, onRemoveItem }) =>
  console.log('B:List') || (
    <ul>
      {list.map((item) => (
        <Item
          key={item.objectID}
          item={item}
          onRemoveItem={onRemoveItem}
        />
      ))}
    </ul>
  );
```

`List` 컴포넌트는 함수 본문(중괄호 블록)이 없고, 개발자들은 게을러서 단순한 로깅 문 하나 때문에 컴포넌트를 리팩터링하고 싶어 하지 않으므로, `||` 연산자를 대신 사용했다. 이는 함수 본문 없는 함수 컴포넌트에 로깅 문을 추가하는 유용한 트릭이다. `||` 연산자 왼쪽의 `console.log()`는 항상 `false`로 평가되므로, 오른쪽 값이 항상 실행(반환)된다.

```js
function getTheTruth() {
  if (console.log('B:List')) {
    return true;
  } else {
    return false;
  }
}

console.log(getTheTruth());
// B:List
// false
```

페이지를 새로고침하면서 브라우저 개발자 도구의 실제 로그에 주목해보자. 먼저 `App` 컴포넌트가 렌더링되고, 이어서 `List` 같은 자식 컴포넌트가 렌더링된다.

```text
// 최초 렌더링
B:App
B:List

// 로딩 표시로 인해 List 대신 App만 렌더링
B:App

// 데이터가 도착한 뒤 다시 렌더링
B:App
B:List
```

> 만약 이보다 더 많은 로그가 찍힌다면, `src/main.jsx` 파일에서 `App` 컴포넌트를 `<StrictMode>`로 감싸고 있지는 않은지 확인해보자. 그렇다면 Strict Mode를 제거하고 로그를 다시 확인해보자. 개발 모드에서 Strict Mode는 구현상의 문제를 감지해서 경고해주기 위해 컴포넌트를 두 번 렌더링하며, 프로덕션 애플리케이션에서는 자동으로 제외된다.

여기까지는 모든 것이 제때 렌더링되므로 문제가 없어 보인다. 이제 `SearchForm` 컴포넌트의 입력 필드에 타이핑을 해보면서 실험을 한 단계 더 진행해보자. 입력한 글자 하나하나마다 다음과 같은 변화를 볼 수 있다.

```text
B:App
B:List
```

**눈에 띄는 점은, `List` 컴포넌트는 리렌더링될 이유가 없는데도 리렌더링된다는 것이다.** 검색 기능은 버튼을 통해서만 실행되므로, `App`을 거쳐 `List`에 전달되는 목록은 키 입력마다 그대로다. 이는 리액트의 기본 동작으로, 어떤 컴포넌트(여기서는 `App`)의 상태가 바뀌면 그 아래의 모든 컴포넌트가 리렌더링되는 것이다. 즉, 부모 컴포넌트가 리렌더링되면 그 자손 컴포넌트들도 함께 리렌더링된다. **리액트는 자식 컴포넌트의 리렌더링을 막는 것이 버그로 이어질 수 있기 때문에 기본적으로 이렇게 동작하며, 이 리렌더링 메커니즘이 기본적으로 충분히 빠르기 때문에 이런 자동 리렌더링 방식이 권장된다.**

### 실습 3. `React.memo`로 불필요한 리렌더링 막기

그럼에도 리렌더링을 막고 싶은 경우가 있다. 예를 들어 테이블에 표시되는 거대한 데이터셋(`List` 컴포넌트)은, 그 데이터에 영향을 주지 않는 갱신(`Search` 컴포넌트)에는 리렌더링되지 않는 편이 낫다. 컴포넌트에 변화가 있었는지 동등성 검사(equality check)를 수행하는 편이 더 효율적이며, 이를 위해 **리액트의 `memo` API로 props에 대한 동등성 검사를 수행할 수 있다.**

```jsx
// src/App.jsx
const List = React.memo(
  ({ list, onRemoveItem }) =>
    console.log('B:List') || (
      <ul>
        {list.map((item) => (
          <Item
            key={item.objectID}
            item={item}
            onRemoveItem={onRemoveItem}
          />
        ))}
      </ul>
    )
);
```

**`React.memo` API는 컴포넌트의 props가 바뀌었는지 검사한다.** 바뀌지 않았다면, 부모 컴포넌트가 리렌더링되더라도 해당 컴포넌트는 리렌더링되지 않는다. 하지만 `SearchForm`의 입력 필드에 타이핑을 해봐도 여전히 같은 로그가 나온다.

```text
B:App
B:List
```

`List` 컴포넌트에 전달되는 `list`는 동일하지만, `onRemoveItem` 콜백 핸들러는 그렇지 않다. `App` 컴포넌트가 리렌더링될 때마다 이 콜백 핸들러도 매번 새로운 함수로 다시 생성되기 때문이다. 앞서 이런 동작을 막기 위해 `useCallback` 훅을 사용해서, 최초 렌더링 시(또는 의존성 중 하나가 바뀔 때만) 함수를 생성하도록 만든 바 있다.

### 실습 4. `useCallback`으로 콜백 핸들러 고정하기

```jsx
// src/App.jsx
const App = () => {
  // ...

  const handleRemoveStory = React.useCallback((item) => {
    dispatchStories({
      type: 'REMOVE_STORY',
      payload: item,
    });
  }, []);

  // ...

  console.log('B:App');

  return ( /* ... */ );
};
```

이 콜백 핸들러는 함수 시그니처의 인자로 `item`을 전달받으므로 별도의 의존성이 없고, `App` 컴포넌트가 처음 렌더링될 때 딱 한 번만 선언된다. 이제 `List`에 전달되는 props는 어느 것도 바뀌지 않아야 한다. **`React.memo`와 `useCallback`을 함께 사용한 상태로 `SearchForm`의 입력 필드에 검색어를 입력해보자. "B:List" 로그는 더 이상 찍히지 않고, `App` 컴포넌트만 "B:App" 로그와 함께 리렌더링된다.**

모든 props가 그대로 유지되더라도, **부모 컴포넌트가 강제로 리렌더링되면 자식 컴포넌트도 함께 리렌더링된다. 이것이 리액트의 기본 동작이며, 리렌더링 메커니즘이 충분히 빠르기 때문에 대부분의 경우 잘 동작한다.** 하지만 리렌더링이 애플리케이션의 성능을 떨어뜨린다면, 리액트의 `memo` API로 리렌더링을 막을 수 있다. 살펴본 것처럼 `memo`만으로는 부족할 때도 있는데, 콜백 핸들러가 부모 컴포넌트에서 매번 새로 정의돼 변경된 props로 전달되면서 또 다른 리렌더링을 유발하기 때문이다. 이럴 때는 `useCallback`을 사용해서, 콜백 핸들러가 의존성이 바뀔 때만 변경되도록 만든다.

## 비용이 큰 연산 다시 실행하지 않기

때때로 컴포넌트의 함수 시그니처와 반환 블록 사이에, 렌더링될 때마다 실행되는 성능 비용이 큰 연산이 있을 수 있다. 이런 상황을 위해 먼저 현재 애플리케이션에 예시 사례를 하나 만들어본다.

### 실습 5. 비용이 큰 연산 예시 만들기

```jsx
// src/App.jsx
const getSumComments = (stories) => {
  console.log('C');

  return stories.data.reduce(
    (result, value) => result + value.num_comments,
    0
  );
};

const App = () => {
  // ...

  const sumComments = getSumComments(stories);

  return (
    <div>
      <h1>My Hacker Stories with {sumComments} comments.</h1>

      {/* ... */}
    </div>
  );
};
```

모든 인자가 함수에 전달된다면, 컴포넌트 내부에서 더 필요한 의존성이 없으므로 컴포넌트 바깥에 함수를 두어도 무방하다. 이렇게 하면 렌더링마다 함수가 새로 생성되는 것을 막을 수 있어 `useCallback` 훅이 필요 없어진다. 하지만 **이 함수는 여전히 렌더링될 때마다 댓글 합계를 다시 계산하는데, 더 비용이 큰 연산이라면 이는 문제가 된다.**

**`SearchForm` 컴포넌트의 입력 필드에 텍스트를 입력할 때마다 이 연산이 다시 실행되며 "C"라는 출력이 나온다.** 지금처럼 무겁지 않은 연산이라면 괜찮을 수 있지만, 만약 이 연산에 500ms 이상 걸린다고 상상해보자. 컴포넌트 안의 모든 것이 이 연산을 기다려야 하므로 리렌더링이 지연될 것이다. 리액트에게 의존성 중 하나가 바뀔 때만 함수를 실행하라고 지시할 수 있는데, 의존성이 바뀌지 않았다면 함수의 결과는 그대로 유지된다. 여기서 리액트의 `useMemo` 훅이 도움이 된다.

### 실습 6. `useMemo`로 연산 결과 캐싱하기

```jsx
// src/App.jsx
const App = () => {
  // ...

  const sumComments = React.useMemo(
    () => getSumComments(stories),
    [stories]
  );

  return ( /* ... */ );
};
```

**이제 `SearchForm`에 타이핑할 때마다 이 연산이 다시 실행되지 않는다. 의존성 배열(여기서는 `stories`)이 바뀔 때만 실행된다.** 결국 이 기법은 컴포넌트의 (리)렌더링을 지연시킬 수 있는, 비용이 큰 연산에만 사용해야 한다.

`useMemo`, `useCallback`, `memo`에 대한 이 시나리오들을 모두 살펴봤으니, **이들을 기본값처럼 습관적으로 사용해서는 안 된다는 점을 꼭 기억하자.** 이런 성능 최적화 기법들은 실제로 성능 병목에 부딪혔을 때만 적용해야 한다. 리액트의 렌더링 메커니즘은 기본적으로 꽤 효율적이기 때문에 대부분의 경우 병목이 발생하지 않는다. **때로는 `memo` 같은 유틸리티의 검사 비용이 리렌더링 자체보다 더 클 수도 있다.**

## 정리

- 리액트는 기본적으로 빠르기 때문에 대부분의 애플리케이션에서는 별도의 성능 최적화가 필요하지 않으며, 이 장은 학습 목적으로 성능 개선 기법을 다룬다.
- `StrictMode`는 부작용 오용 같은 잠재적 버그를 표면화하기 위해, 개발 환경에서 `useEffect` 훅을 의도적으로 두 번 실행시킨다. 개발 환경에만 적용되고 프로덕션 빌드에서는 자동으로 제거된다.
- `useRef`의 `current` 프로퍼티는 리렌더링을 유발하지 않고도 값을 유지할 수 있어서, 첫 렌더링에서는 부작용 함수 호출을 건너뛰고 이후 리렌더링에서만 실행하는 패턴을 구현할 수 있다.
- 부모 컴포넌트가 리렌더링되면 자식 컴포넌트도 기본적으로 함께 리렌더링되며, `React.memo`는 props가 바뀌지 않았을 때 이 리렌더링을 건너뛰게 해준다.
- `React.memo`만으로는 콜백 함수 props가 매번 새로 생성되는 문제를 막을 수 없으며, `useCallback`과 함께 사용해야 자식 컴포넌트의 불필요한 리렌더링을 완전히 방지할 수 있다.
- 렌더링마다 반복 실행되는 비용이 큰 연산은 `useMemo` 훅으로 감싸서, 지정한 의존성이 바뀔 때만 다시 계산되도록 캐싱할 수 있다.
- `useMemo`, `useCallback`, `React.memo`는 모두 실제 성능 병목이 확인됐을 때만 사용해야 하는 최적화 기법이며, 습관적으로 남용하면 오히려 검사 비용이 이점을 상회할 수도 있다.
