# 복잡한 문제를 다루는 API를 설계하는 방법 I

이 단계에서는 TDD의 핵심 원칙인 "테스트를 먼저 작성하고 이를 기준으로 설계를 확장해 나가는" 방법을 다룬다. 복잡한 문제를 작은 테스트로 축소하여 시작하는 방법과, 현실 세계의 개념을 메타포로 치환하여 객체 지향적인 도메인 모델을 설계하는 방법을 체험한다.

## 개념

### 복잡한 문제는 작은 테스트로 축소하여 시작하라!

복잡한 목표를 어떻게 테스트 코드로 표현해야 할 지 확신이 서지 않을 때는, 
큰 문제를 단순화하여 **작은 테스트로 시작**하고, 점진적으로 진척을 만들어 나가는 방법을 사용해야 한다. 예를 들어, 다중 통화 덧셈($5 + 10 CHF = $10)이라는 복잡한 기능을 당장 구현하는 대신, 해결하기 쉬운 동일 통화의 단순한 덧셈 테스트($5 + $5 = $10)부터 작성하며 점진적인 진척을 만들어 낸다.

## 실습

> $5 + 10 CHF = $10 이라는 다중 통화 덧셈을 구현하기 전에 먼저 동일 통화의 단순한 덧셈($5 + $5 = $10)을 먼저 검증한다.

### Red - 동일 통화 덧셈 API를 먼저 설계

혼합 통화의 덧셈($5 + 10 CHF)을 테스트하는 대신 가장 단순한 형태의 같은 통화의 덧셈($5 + $5 = $10)을 검증하는 테스트를 작성하며 시작한다.

**테스트 코드:**
```java
@Test
public void testSimpleAddition() {
  Money sum = Money.dollar(5).plus(Money.dollar(5));
  assertEquals(Money.dollar(10), sum);
}
```
- $5 + $5 = $10 을 검증하는 테스트를 추가한다.

**프로덕션 코드:**
```java
Money plus(Money addend) {
    return null; // 컴파일 통과를 위해 임시로 null 반환
}
```
- plus() 메서드를 임시로 구현하여 컴파일을 통과시킨다.
- 아직 plus()가 실제로 덧셈을 수행하는 것은 아니기 때문에 테스트는 실패한다.


### Green - 동일 통화 덧셈 API 구현

plus() 메서드를 실제로 덧셈을 수행하도록 구현하여 테스트를 통과시킨다.

```java
// Money 클래스:
Money plus(Money addend) {
  return new Money(amount + addend.amount, currency);
}
```
- plus() 메서드가 실제로 덧셈을 수행하도록 구현한다.
