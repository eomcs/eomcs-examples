# TDD 이해하기: Red-Green-Refactor 사이클

이 단계에서는 테스트 주도 개발의 핵심인 Red-Green-Refactor 사이클을 체험하는 것이 목표다.

**TDD 작업 방식:**
- 작은 테스트를 먼저 작성하고, 그 테스트가 실패하는 것을 확인한 후, (Red 단계)
- 최소한의 코드를 작성하여 테스트가 통과하도록 만드는 과정을 반복한다. (Green 단계)
- 마지막으로, 중복을 제거하고 코드를 개선하는 리팩토링 단계로 넘어간다. (Refactor 단계)

**테스트 코드를 작성하는 과정**은 단순한 '오류 검증'을 넘어 <u>개발자의 설계 의도(Design Intent)와 이상적인 인터페이스를 가장 먼저 표현하고 구체화하는</u> 핵심적인 **설계 활동**이다.

켄트 벡은 **테스트 코드가 설계 의도를 표현하는 수단**이라는 점을 다음과 같은 맥락에서 설명한다.
- **완벽한 인터페이스 상상하기:** 
테스트를 작성한다는 것은, 해당 연산이 외부에서 어떻게 보여야 할지에 대한 '이야기(story)'를 쓰는 것이다. 처음부터 복잡하고 현실적인 제약을 따지기보다는, <u>자신이 사용하고 싶은 가장 완벽한 형태의 API를 상상</u>하여 코드로 표현한 뒤, <u>이를 현실로 만들기 위해 역방향으로 작업</u>해 나가는 것이 TDD의 방식이다.
- **설계적 판단을 테스트로 번역하기:** 
개발자가 코드를 보며 느끼는 설계적 불만이나 미적 판단(예: "연산을 수행했을 때 기존 객체의 상태가 변하는 부작용(side effect)이 있어서는 안 된다")을 구체적인 테스트 케이스로 번역하는 것은 TDD의 핵심적인 과정이다. 이를 통해 <u>모호하고 추상적인 설계 토론을 피하고, 구체적인 사례를 바탕으로 시스템의 동작 방식을 명확히 설계</u>할 수 있다.
- **동시다발적인 설계 문제 해결:** 
단지 하나의 테스트를 작성하는 순간에도 개발자는 <u>여러 설계 문제들을 동시에 고민하고 결정</u>하게 된다. 이 기능이 어느 객체에 속해야 하는지, 새로운 클래스를 만들어야 할지 기존 메서드를 수정할지, 메서드의 이름은 무엇으로 할지 등의 <u>설계 의도가 테스트 코드의 형태를 통해 고스란히 결정</u>된다.
- **메타포(Metaphor)를 통한 객체의 책임 분리:** 
복잡한 문제를 그대로 코드로 옮기려 하면 하나의 객체가 여러 역할을 떠안게 되어 책임이 뒤섞이기 쉽다. TDD에서는 이러한 복잡성을 직접 다루기보다, <u>문제를 현실 세계의 개념(메타포)으로 치환하여 이해</u>하려 한다. 예를 들어 서로 다른 통화를 더하고 환율을 적용하는 문제를 단순한 계산 로직이 아니라 ‘은행(Bank)’과 ‘수식(Expression)’이라는 개념으로 바라보는 순간, 각 객체가 수행해야 할 책임이 자연스럽게 분리된다. 수식(Expression)은 “무엇을 계산할 것인가”를 표현하는 역할을 맡고, 은행(Bank)은 “어떻게 계산할 것인가(환율 적용 및 결과 변환)”를 담당한다. 이처럼 메타포는 객체의 역할을 인위적으로 나누는 것이 아니라, 우리가 이미 익숙한 현실의 개념을 통해 책임을 자연스럽게 드러내는 설계 도구가 된다. 이런 메타포(예: Bank, Expression)는 머릿속 아이디어로만 두면 쉽게 흐려진다. <u>테스트 코드는 이 메타포를 실행 가능한 형태로 먼저 선언해 고정</u>하는 역할을 한다. 

즉, TDD에서 테스트를 먼저 작성한다는 것은 단순히 시스템이 고장 나지 않았음을 증명하려는 것이 아니다. 이는 **"내가 만들고자 하는 객체가 외부 세계와 어떤 인터페이스로 협력해야 하는가"에 대한 개발자의 목표와 설계 의도를, 컴퓨터가 실행하고 검증할 수 있는 형태의 코드로 가장 먼저 기록하는 창조적인 설계 행위**라고 할 수 있다.


## 실습

> `$5 × 2 = $10`을 검증하는 예제로 TDD 사이클을 경험해보자.

### Red 단계

`$5 × 2 = $10`을 검증하는 테스트 코드를 먼저 작성한다. 테스트 코드에는  작성할 때 핵심은 어떤 식으로 동작해야 하는지를 설계하여 테스트 코드로 표현하는 것이다. 아직 `Dollar` 클래스가 존재하지 않으므로 컴파일 자체가 되지 않는 상태이며, 이것이 TDD의 **Red** 상태다.

### Green 단계

컴파일이 되고 테스트가 통과하도록 최소한의 코드를 작성한다. 우선 `Dollar` 클래스를 만들어 컴파일이 되게 하고, 처음에는 `amount = 10` 하드코딩도 허용하여 테스트를 통과시킨다. 이것이 TDD의 **Green** 상태다.

### Refactor 단계

하드코딩된 부분을 일반화하여 테스트와 코드 사이의 논리적 중복을 제거한다.`DollarTest`의 `5 * 2 = 10` 테스트와 `Dollar`의 `amount = 10` 코드는 논리적 중복이며, 이 중복을 제거함으로써 두 코드 사이의 의존 관계를 없앤다. 이것이 TDD의 **Refactor** 상태다.
