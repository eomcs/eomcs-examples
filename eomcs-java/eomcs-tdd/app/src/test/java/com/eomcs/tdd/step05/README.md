# 테스트 코드와 실제 코드 간의 결합도 낮춰라!

이 단계에서는 테스트 코드에서 객체의 내부 필드에 직접 접근할 때 발생할 수 있는 문제점을 인식하고 이를 회피하기 위한 방법을 알아본다. equals() 메서드를 활용하여 테스트 코드와 실제 코드 간의 결합도를 낮추는 방법을 학습한다. 또한, 인스턴스 변수를 안전하게 은닉하기 위해 private 을 활용하는 방법도 다룬다.

## 개념

### 결합도

결합도란 시스템의 한 부분을 변경할 때 다른 부분도 함께 변경해야만 하는 종속적인 상태를 의미한다. 스티브 프리먼(Steve Freeman)은 코드와 테스트 사이의 근본적인 문제는 중복 그 자체라기보다는, **"하나를 변경하지 않고는 다른 하나를 변경할 수 없는 의존성"**에 있다고 지적했다. 결합도가 높으면 코드의 유연성이 떨어져 시스템 변경과 확장이 매우 어려워진다.

### 결합도가 높을 때 발생할 수 있는 문제
- **변경의 파급 효과:** 예를 들어, 특정 데이터베이스 벤더의 SQL 구현 세부 사항이 코드 곳곳에 흩어져 있다면(결합도가 높다면), 다른 데이터베이스로 변경하기 위해 관련된 모든 코드를 찾아 수정해야 하는 큰 비용이 발생한다.
- **테스트와 코드의 결합 문제:** 테스트 코드가 실제 코드의 내부 구현에 깊이 의존하고 있다면, 실제 코드의 내부 구조를 조금만 수정해도 관련된 모든 테스트가 깨지게 되어 테스트 코드를 함께 수정해야 하는 부담이 생긴다.


### 문제의 테스트 코드
```java
// 기존의 `testMultiplication()`은 `times()`의 결과를 다음처럼 검증한다.
Dollar product = five.times(2);
assertEquals(10, product.amount);  // amount 필드에 직접 접근!
```
- 테스트가 객체 내부 구현에 결합(coupling)되어 있다.
- 테스트 코드가 객체 내부를 들여다볼수록 캡슐화가 약해진다. 
- 즉 실제 코드의 변경과 유지보수가 어려워진다.


### 결합도를 낮춘 테스트 코드
```java
// `equals()`를 활용하여 Dollar 객체끼리 직접 비교한다.
assertEquals(new Dollar(10), five.times(2));  // 내부 필드 접근 불필요
```
- 테스트가 더 이상 `amount`를 직접 참조하지 않으므로 테스트 코드와 실제 코드의 결합도를 낮춘다.
- equals()처럼 공개된 인터페이스를 통해 검증하면 내부 구현을 자유롭게 숨길 수 있다.


## 실습

> 객체 필드에 직접 접근하는 코드 제거

### Refactor 단계

`testMultiplication()`이 `product.amount`에 직접 접근하므로 테스트가 내부 구현에 강하게 결합되어 있다. `amount`가 package-private이라 테스트에서 접근은 가능하지만, 캡슐화가 약한 문제가 있다.

`testMultiplication()`에서 `product.amount` 직접 접근을 제거하고,
이전 단계에서 구현한 `equals()`를 이용해 `assertEquals(new Dollar(10), five.times(2))` 형태로 리팩토링한다. 테스트가 내부 필드를 더 이상 참조하지 않으므로 `amount`를 `private`으로 선언할 수 있게 되어 캡슐화가 완성된다.

**테스트 코드:**

```
testMultiplication():
  - `product.amount` 직접 접근 제거
```

**프로덕션 코드:**
```
Dollar` 클래스:
  - `amount` 필드를 `private`으로 변경
```
