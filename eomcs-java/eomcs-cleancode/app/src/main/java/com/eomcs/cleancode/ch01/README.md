# Chapter 1. Clean Code (클린 코드란 무엇인가)

## 1. 학습 목표

- 클린 코드의 필요성을 이해한다
- 더러운 코드가 시스템에 미치는 영향을 설명할 수 있다
- 클린 코드의 특징을 정의할 수 있다
- 개발자의 책임과 태도를 이해한다

---

## 2. 왜 클린 코드가 중요한가?

**프로그래밍이란?**

> **기계가 실행할 정도로 상세하게 요구사항을 명시하는 작업**이다. 이렇게 명시한 결과가 바로 **코드**다.

> **DSL(Domain-Specific Language)**: 특정 문제 도메인, 특정 문제 표현 기법, 특정 문제 해결 기법에 사용할 목적으로 만든 프로그래밍 언어나 명세 언어를 의미한다. 해당 분야 전문가들의 의사소통을 돕기 위해, 모호함을 없애며 표현력을 높인 특징이 있다.

**더러운 코드의 문제:**

- 생산성 저하
- 유지보수 비용 증가
- 작은 수정도 어려워짐
- 버그 증가
- 시스템 전체의 품질 저하
- 사례:
    - 한 줄 수정 → 수백 군데 수정
    - 변경이 두려워짐 → 코드 방치
    - 결국 코드가 "썩어감"

**생산성 vs 시간:**

- 잘못된 믿음:
    - "빨리 개발하려면 코드를 대충 작성해야 한다"
- 실제 진실:
    - "코드를 깨끗하게 유지하는 것이 가장 빠른 개발 방법이다"
    - 더러운 코드 → 즉각적인 속도 저하
    - 클린 코드 → 지속적인 속도 유지

**대규모 재작성의 실패**

- 흔한 패턴:
    1. 기존 코드가 너무 더러움
    2. 새로 만들기로 결정
    3. 새로운 팀 구성
    4. 기존 시스템 유지 + 신규 개발 병행
- 결과:
    - 일정 지연
    - 두 시스템 모두 유지 부담 증가
    - 새 시스템도 결국 더러워짐

**르블랑의 법칙(LeBlanc's law)**: 

> 개발자들은 엉망인 코드를 보며 "나중에 정리해야지"라고 위안하지만, **"나중은 결코 오지 않는다"**

---

## 3. 책임은 누구에게 있는가?

**흔한 핑계:**

- 요구사항이 문제다
- 일정이 너무 짧다
- 매니저가 문제다

**진실:**

- 문제의 책임은 개발자에게 있다
- 개발자는 기술적 판단을 해야 한다
- 잘못된 방향에 대해 NO라고 말해야 한다

**의사 비유:**

- 환자: "손 씻지 말고 빨리 수술해 주세요"
- 의사: 거절해야 함

**개발자도 동일:**

- 비전문적인 요구를 그대로 따르는 것은 비전문적인 행동이다

---

## 4. 클린 코드란 무엇인가?

여러 유명 개발자의 공통 의견:

> "우아하고 효율적이며, 논리가 직관적이어서 버그가 숨어들지 못하는 코드. 또한 한 가지를 잘하는 코드여야 한다.", **비야네 스트롭스트룹 (Bjarne Stroustrup)**

> "단순하고 직접적이며, 잘 쓴 산문처럼 읽히는 코드. 설계자의 의도를 숨기지 않는 코드다.", **그래디 부치 (Grady Booch)**

> "작성자가 아닌 다른 사람도 읽기 쉽고 고치기 쉬운 코드. 무엇보다 단위 테스트와 인수 테스트가 존재해야 한다.", **데이브 토마스 (Dave Thomas)**

> "누군가 주의 깊게 짰다는 느낌을 주는, 즉 코드를 아끼고 돌보는(care) 사람이 작성한 코드다.", **마이클 페더스 (Michael Feathers)**

> "모든 테스트를 통과하고, 중복이 없으며, 설계 아이디어를 잘 표현하고, 클래스와 메서드를 최소화한 코드다.", **론 제프리스 (Ron Jeffries)**

> "코드를 읽을 때 당신이 짐작했던 그대로 수행되는 코드. 언어를 탓하지 않고, 언어를 단순하게 보이도록 만드는 코드다.", **워드 커닝햄 (Ward Cunningham)**

**클린 코드의 특징:**

1. 읽기 쉬운 코드 (Readability)
    - 다른 개발자가 쉽게 이해 가능
    - 코드 자체가 설명이 된다
2. 단순한 코드 (Simplicity)
    - 불필요한 요소 제거
    - 복잡성 최소화
3. 한 가지 일만 하는 코드
    - 함수와 클래스는 하나의 책임만 가진다
4. 변경하기 쉬운 코드 (Maintainability)
    - 수정이 쉽다
    - 확장이 쉽다
5. 테스트 가능한 코드
    - 단위 테스트 존재
    - 안정적인 변경 가능
    - 테스트 없는 코드는 클린 코드가 아니다
6. 작은 코드 (Smallness)
    - 코드의 크기는 작을수록 좋다
    - 함수, 클래스, 모듈 모두 해당
7. 배려가 느껴지는 코드
    - 의도가 명확함
    - 개선할 부분이 거의 없음

---

## 5. 코드 감각 (Code Sense)

**정의:**

> 좋은 코드와 나쁜 코드를 구분하는 능력

**특징:**

- 코드의 문제를 직관적으로 인지
- 더 나은 구조를 떠올림
- 점진적으로 개선 가능

**중요한 점:**

- 타고나는 것이 아니라 학습 가능
- 경험과 반복을 통해 향상됨

---

## 6. 더러운 코드의 본질

- 읽기 어려움
- 중복 코드 많음
- 의도가 불명확
- 변경 시 리스크 증가

---

## 7. 깨끗한 코드의 본질

- 읽기 쉬움
    - 기존 코드를 읽어야 새 코드를 짤 수 있다.
    - 읽기 쉽게 만들면 짜기도 쉬워진다.
- 명확한 의도
- 변경 용이
- 지속적인 개선 가능

---

## 8. 핵심 메시지

**가장 중요한 문장:**

> "빠르게 가는 유일한 방법은 코드를 깨끗하게 유지하는 것이다"

**개발자의 역할:**

- 코드 품질을 지키는 책임자
- 기술적 판단을 내리는 전문가
- 장인정신(Craftsmanship)을 가진 사람

---

## 9. 토론 질문

1. 왜 개발자는 더러운 코드를 작성하게 되는가?
2. 일정 압박 상황에서 클린 코드를 유지할 수 있는 방법은?
3. "테스트 없는 코드는 클린 코드가 아니다"에 동의하는가?
4. 코드 감각(Code Sense)은 어떻게 훈련할 수 있는가?

---

## 10. 정리

- 클린 코드는 선택이 아니라 필수
- 더러운 코드는 반드시 시스템을 망친다
- 속도를 위해 클린 코드를 포기하면 더 느려진다
- 개발자는 코드 품질에 대한 책임이 있다
- 클린 코드는 기술이 아니라 습관이며 태도이다
